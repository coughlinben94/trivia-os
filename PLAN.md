# Plan: Nightly Round-Journey Designer Agent ("Storybook Agent")
_Locked via grill — by Claude + Ben, 2026-07-20. Revised through Codex review rounds 1–3; APPROVED round 4. Revised through Gemini cross-review round 1; APPROVED round 2. Fully converged, both models._

## Goal

An autonomous overnight agent that designs one round-journey prototype per night for Trivia OS. Every journey is an animated storybook page: something happens, has a consequence, gets resolved (beginning/middle/end beats are mandatory in every brief). A single design is not necessarily a one-night job — the agent can carry a piece across multiple nights, refining it against Ben's own concrete notes until it's right, rather than treating every session as a blank slate. The agent executes pre-grilled theme briefs from a queue, generates vector sprites via the validated Recraft recipe, builds a standalone HTML prototype to the full `references/round-journeys.md` conventions, runs a static code-invariant self-audit, and commits results to `concepts/` for Ben's morning review in a local, double-clickable gallery. Ben's eyes remain the only approval gate — the agent never wires anything into the live app; approved drafts graduate through the real, already-documented `SlideRenderer.jsx` promotion path, by Ben's own hand.

## Approach

1. **Shared scaffolding in repo (runtime-agnostic):**
   - `concepts/AGENT-PROMPT.md` — full nightly instructions (the agent's brain; versioned, editable)
   - `concepts/QUEUE.md` — theme entries, each with an immutable id and status machine `raw → grilled → building → built → { approved | rejected | needs-revision }`. A `grilled` entry carries a locked one-paragraph brief with explicit beginning/middle/end story beats. Cross-theme handoff briefs name both themes. **`needs-revision` is distinct from `rejected`:** it means Ben watched the draft, liked the direction, and gave one or more concrete fixable notes (per round-journeys.md's feedback protocol) — not "abandon," but "keep working on this specific piece." An entry may also carry `targetShow`/`targetRound` (optional) pointing at a real upcoming show/round boundary, so a journey isn't just an abstract art object but explicitly slated for a real Friday night when Ben knows which one.
   - `concepts/LESSONS.md` — Ben's rejection/approval notes. **Capped, not append-forever:** the file keeps only the 10 most recent active directives; on the 11th, the agent (during a normal run, before building) folds the oldest entries into one summarized "established conventions" block at the top and drops their verbatim originals. Injected into every nightly run at whatever size it currently is — never unbounded.
   - `concepts/manifest.js` — a JS global (`window.MANIFEST = [...]`; same pattern as the fact-feed's `facts.js`) so the gallery works from `file://` with zero server. Entry shape: `{ id, file (nullable when blocked), theme, journeyType: 'same-theme'|'cross-theme', fromTheme?, toTheme?, targetShow?, targetRound?, status: 'draft'|'blocked'|'needs-revision'|'rejected'|'approved'|'shipped', degraded: bool, source: 'ben-grilled'|'agent-proposed+self-grilled', date, iteration: n, supersedes: <previous file for this id, or null>, revisionNotes: [{ note, date }], sprites: [{ prompt, model, date }] }`. **Iterations are never overwritten in place** — each pass on the same queue id writes a new file (`<slug>-v<n>.html`) and a new manifest entry linked via `supersedes`, so the gallery's side-by-side compare can show v1 next to v2 next to v3 without losing history to a silent overwrite.
   - `concepts/NIGHTLY-LOG.md` — per-run log: what built, generation calls spent, blockers, audit summary
   - `concepts/tools/` — self-contained toolbox with its own `package.json`/`node_modules` (installed once at build-out, inside `concepts/` — root package files never touched): `sanitize-svg.mjs` (parser-based allowlist sanitizer using a real XML parser dependency, never regex, configured to reject `<!ENTITY` declarations outright and enforce a max byte-size + max node-depth ceiling before parsing — closes entity-expansion/billion-laughs and oversized-tree denial-of-service, not just script injection) + adversarial fixtures (including entity-expansion and depth-bomb cases) + committed `package-lock.json`; preflight verifies sanitizer deps and runs `npm ci --prefix concepts/tools` when absent
   - `concepts/.nightly-lock` — transient run lock (runId, session identity, start, 4h expiry)
   - `concepts/index.html` — static gallery: iframes prototypes (`sandbox="allow-scripts"`), lazy-loads and pauses offscreen frames, Replay, side-by-side compare, reads `manifest.js`. Opens by double-click; no server, no deploy. Morning front door. Degraded drafts visibly badged.

2. **Dual trigger, same brain, same lock:**
   - Primary: Cowork scheduled task, nightly 1:00 AM — prompt is "read `concepts/AGENT-PROMPT.md` and follow it." Recraft MCP connector available (scheduled tasks run inside a Claude surface).
   - Secondary: manual Claude Code run (`claude -p "$(cat concepts/AGENT-PROMPT.md)"`) for daytime tuning/extra runs; Recraft via `claude mcp add --transport http recraft https://mcp.recraft.ai/mcp` (one-time OAuth).
   - **Both triggers run the identical preflight, including `.nightly-lock` acquisition** (step 3a below) — a daytime manual run that finds a live lock exits immediately, same as a second scheduled run would. There is no separate, lighter-weight manual path; the lock is what makes "dual trigger" safe rather than a race condition waiting to happen.

3. **Nightly run sequence (encoded in AGENT-PROMPT.md):**
   a. **Preflight (fail fast, before any claim or spend):** repo present; `QUEUE.md`/`LESSONS.md`/`references/round-journeys.md` readable; Recraft reachable (`get_user`); `git fetch origin` + record current branch/auth state; record the dirty-file **baseline** as content hashes (`git status --porcelain` paths + SHA-256 of each out-of-scope dirty/untracked file — hash comparison, not porcelain-line comparison, so a re-modified already-dirty file is still caught); acquire `concepts/.nightly-lock` atomically (runId, session identity, start time, 4h expiry — if a live lock exists, exit; a lock past expiry may be cleared with a log note — expiry takeover is the one explicit exception to the owner-only-classifies rule). Stale `building` queue entries whose lock is expired → mark `blocked` with note. Any preflight failure → skip-and-log, zero spend.
   b. Read `AGENT-PROMPT.md`, `QUEUE.md`, `LESSONS.md`, `references/round-journeys.md`; load `gsap-core`, `gsap-timeline`, `emil-design-eng` before building and `impeccable` for the audit (mandatory, per round-journeys.md). If the journey touches an existing theme: also read `client/src/themes/index.js` tokens and that theme's `ParticleBackground.jsx` scene — the full engine file, not a token extract: round-journeys.md's own input-requirements section mandates the real existing engine so a motif extends real code rather than an invented generic shape, and this is a read-only reference load, not something written back to app source, so its token cost is a one-time per-run expense, not a growing liability.
   c. **Claim (priority order — a stalled draft always wins over starting something new):** (1) the oldest `needs-revision` entry, if any — this is a multi-night continuation, not a fresh build: read the previous iteration's file (`supersedes` chain) and its `revisionNotes` in full before touching anything, and produce the *next* numbered iteration of the *same* id, never a from-scratch rebuild of a piece Ben already approved the direction of. (2) else the oldest `grilled` entry. (3) else invent a fresh theme concept from scratch (NOT from the existing 21 — new world, own motif) — the brief must define concrete palette hex tokens (real material, not descriptions) — then **self-grill** it against the five-part scene checklist + motion-technique checklist, writing the Q&A brief before building. Label `agent-proposed + self-grilled`; these are speculative concepts whose eventual promotion explicitly includes human theme-creation work. **Iteration cap: 5 passes per id.** On the 5th `needs-revision` cycle without approval, the agent does not attempt a 6th — it logs "iteration cap hit, needs your call" and falls through to priority (2)/(3) for that night, leaving the capped entry exactly as it was for Ben to decide by hand (keep manually iterating, or mark `rejected`). This is the same class of guard as the nightly credit cap: bounds one stuck idea from silently eating every night indefinitely. Whichever entry is claimed: set it to `building`, write the opening NIGHTLY-LOG entry, and **commit + push this claim immediately** (`nightly: claim <id>`) via the same single guarded-push routine used at completion (fetch → divergence check → baseline check → explicit push; skip-and-log on any guard trip) — this commit is the crash checkpoint; the finished prototype is a second commit through the same routine.
   d. Generate 2–4 vector sprites via the validated Recraft recipe: `generate_image`, `input_style: icon`, `model: recraftv2`, short (<25 word) noun-first prompts, one shared color-language phrase repeated verbatim across the set, SVG pulled directly. **Cap: 10 generation calls/night, counted per call.** The full sprite set for the night's one theme is generated as a single batch before any HTML is written. On two consecutive failures, or the cap being reached before the batch completes: discard the partial batch and downgrade the **entire** prototype to hand-drawn SVG primitives — never a mix of Recraft sprites and primitives in the same scene, which would look like two different shows collided. Set `degraded: true`, flag in notes. Every sprite's SVG passes the committed parser-based sanitizer (`concepts/tools/sanitize-svg.mjs` — element/attribute allowlist, strips scripts/event handlers/foreignObject/external refs/CSS url()s/dangerous hrefs; fail closed; ships with adversarial fixtures that must be rejected during the supervised first run) and is **embedded inline** in the prototype (or committed as a local file) — external URLs (e.g. img.recraft.ai) are an audit failure. Provenance (prompt, model, date) recorded per sprite in manifest + notes.
   e. Build the prototype: single self-contained HTML in `concepts/`, real 16:9 stage, real palette values, Replay button, working reduced-motion simulate toggle, GPU-only animation. **Two legal mechanisms, chosen by piece, not mixed within one:** GSAP timelines (loaded per round-journeys.md's mandatory `gsap-core`/`gsap-timeline` skills) use GSAP's own `gsap.ticker.lagSmoothing()` and visibility handling — never hand-roll frame-timing math on top of GSAP, that fights its internal scheduler. Genuinely per-frame canvas/rAF pieces (particle fields, warp-engine extensions) use the existing app convention verbatim — `dtn = min(26, dt)/16.667` (copied from `TeamPickerSlide.jsx` per round-journeys.md's own "Reusable lessons," not reinvented) plus a clock reset on visibility change/replay, notes block, self-gate block, and the gallery **postMessage contract** (play/pause/replay; validated by exact `event.source` identity + per-frame nonce + strict message schema — origin alone is NOT trusted, since `file://` frames report the opaque origin `"null"`; nonce delivered by parent via URL fragment set before `src` assignment; pause = cancel rAF + a document-level class pausing `animation-play-state` on all animated descendants, not just the root; replay = clock reset). One prototype per night, one file. A cross-theme brief yields one prototype containing departure flourish + neutral bridge + arrival flourish per round-journeys.md.
   f. **Static self-audit and fix before commit** (code-invariant verification — honest scope: no browser in this runtime): contrast-ratio math, timing sums vs. notes text, frame-rate independence (dtn present + visibility reset), GPU-only compliance, silhouette check against every declared background state, particle pooling (no per-frame allocation), near-black banding mitigation present, no external references, sanitizer-passed SVG only, postMessage contract present, reduced-motion branch verified by code inspection. Audit summary written into the notes block. Runtime feel, TV banding, and motion-smoothing checks remain Ben's morning review + venue check, explicitly.
   g. Update queue entry (`built` or `blocked` — a completed revision pass clears `needs-revision` back to `built` for fresh review, it does not jump straight to `approved`), `manifest.js` (new entry gets `iteration: (previous iteration + 1)` and `supersedes: <previous file>` when this run was a revision pass, else `iteration: 1` and `supersedes: null`) (written ONLY as `window.MANIFEST = <JSON.stringify output>;` — single assignment, no expressions; gallery renders all fields via `textContent`; **written atomically** — full content to `manifest.js.tmp` first, then renamed over the live file, so a crash or session timeout mid-write can never leave a truncated file that bricks the gallery), `NIGHTLY-LOG.md`.
   h. **Commit with guards:** stage an explicit allowlist of files this run created/changed (never blanket `git add concepts/`); if `git status` shows delta from the preflight baseline outside `concepts/`, abort commit and log. Message prefix `nightly:`. **Push:** re-fetch, re-check divergence immediately before an explicit `git push origin main:main`; skip and log if `origin/main..main` contains non-`nightly:` commits (Ben's unpushed day work is never published); non-fast-forward = logged safe failure, never force. The lock is held through validation, commit, push-or-safe-failure logging, and cleanup — released only as the final step, even on failure paths. No wiring into `SlideRenderer.jsx`, ever.

4. **Stuck protocol:** any wall (Recraft dead even for fallback decisions, brief too vague, unfixable audit contradiction, preflight failure) → skip-and-log: NIGHTLY-LOG entry (what blocked, what was tried, what's needed from Ben), queue/manifest status `blocked`, clean exit. Never guess through ambiguity.

5. **Morning loop — three outcomes, not two:** Ben double-clicks the gallery, replays drafts (and, for anything past iteration 1, the version history via `supersedes` — side-by-side against the prior pass), and gives one of three verbal calls **in a daytime Claude session** — Claude updates manifest/queue status accordingly (Ben never hand-edits JSON):
   - **Approve:** direction and execution are both right. Status → `approved`. Ben also gives one line on *why it landed* — Claude appends it to `LESSONS.md`'s Established Conventions as a positive pattern, not just a caution. Without this, the pipeline only ever learns what to avoid (from rejections) and never what to repeat — a system fed exclusively on don'ts drifts toward safe, generic work over time. Graduates to the real, human-driven promotion path — see "Promotion path" below.
   - **Reject:** wrong direction entirely, not worth continuing. Status → `rejected`. Concrete reason appended to `LESSONS.md` (general, cross-theme takeaway) so future *different* themes don't repeat the same mistake.
   - **Needs revision:** right direction, specific fixable problem(s). Status → `needs-revision`, with Ben's concrete note(s) (per round-journeys.md's feedback protocol — name the specific thing, not a vibe) written into that queue entry's `revisionNotes`, not the general LESSONS.md — these notes are local to this one piece, not a studio-wide lesson. Next run picks this entry up first (see §3c priority order) and produces the next numbered iteration from the previous file, not a rebuild.

   Degraded drafts can't be approved until sprites are regenerated, regardless of which of the three calls Ben makes.

   **Promotion path (approved drafts only) — grounded in the actual app, not abstract:** per `PLAN-round-journey-studio.md` §2's three real steps — (1) new component file in `client/src/components/display/slides/`, following the existing theatrical-canvas/rAF precedent (`TeamPickerSlide.jsx`, `WinnerRevealSlide.jsx`), built by Ben's own hand from the approved prototype's logic; (2) import + one-line registration in `SlideRenderer.jsx`'s `SLIDE_COMPONENTS` map, plus a `SLIDE_ANIMATIONS` entry only if custom content-entrance timing is needed beyond the defaults; (3) surfaced in the host build/live-control surface so a real round in a real show can be told to use it. Whether a given journey becomes a `round-intro` variant or a dedicated `round-journey` slide type is explicitly still an open call per that doc — "once we have 2-3 shipped journeys, the pattern will be obvious" — this pipeline's job is to get real approved drafts to that decision point, not to pre-decide it. This agent never performs any of these three steps itself; they are 100% Ben, by hand, exactly as already documented.

6. **Build order:** (1) gallery + manifest.js, (2) QUEUE/LESSONS/NIGHTLY-LOG skeletons, (3) AGENT-PROMPT.md, (4) scheduled task creation, (5) one supervised manual run with Ben watching, fix breakage, then unattended.

## Key decisions & tradeoffs

- **Cowork scheduled task as primary trigger** (not cron/headless): MCP connectors (Recraft) only run inside Claude surfaces. Claude Code kept as manual second trigger off the same prompt file — zero duplication, prompt tuning benefits both.
- **Gallery is local-file-first by design** (manifest.js global, not fetched JSON): double-click works, no server, no deploy dependency. Vercel/phone review dropped from scope — the Vite root is `client/` and `concepts/` isn't served; solving that buys little for a one-reviewer pipeline.
- **Sprites generated in-loop nightly** (not deferred to review time): morning drafts must not look AI-crude — that complaint is what recruited Recraft. Sanitized, inlined, provenance-tracked; fallback primitives keep the run alive through API flakiness (500s observed 2026-07-19/20), visibly marked degraded.
- **Agent commits + pushes with guards** (confined to `concepts/`, `nightly:` prefix, push skipped when Ben's unpushed work is present): versioned drafts without ever publishing unrelated work. Worktree/PR/scoped-credential infra rejected as overhead for this scale.
- **Grill gate on every theme**: no build without a grill doc. Ben-fed themes are grilled live (Ben + Claude, daytime). Agent-proposed themes (queue-empty nights) are self-grilled solo and labeled as such — pipeline stays warm, discipline stays written.
- **Fresh themes, not the existing 21**: agent proposals invent new worlds (Ben's explicit call — the 21 are the app's past). Mitigation for the real-material rule: proposed briefs must define concrete hex palettes, and these concepts are labeled speculative with a longer promotion path.
- **One prototype/night, 10-call cap**: quality over volume ("designing forty and shipping two" is the named failure mode); cap counted in generation calls, not opaque credits.
- **Human approval is the gate by process, not by cryptography**: the agent cannot mark `approved`, cannot touch `client/src`, cannot register slide types (prompt rules + staged-paths guard). Protected-branch/signed-commit enforcement rejected as process theater for an audience of one — the real gate is that nothing reaches the app without Ben hand-promoting it.

## Risks / open questions

- Recraft `create_style` (true style-lock) is currently 500ing platform-wide; recipe relies on repeated color-phrase matching, which held for a 3-sprite navy/orange set but is unproven on larger/subtler palettes. Retry style-lock periodically.
- Long-run truncation: the claim-first checkpoint (queue `building` + log entry) means a killed run is detected and marked `blocked` at next preflight — but work after the last commit is lost. Accepted; prompt orders the expensive steps late.
- Self-grilled agent briefs are the quality gamble by design; LESSONS.md is the corrective loop. First 3–5 runs need prompt tuning — expected, not failure.
- Silent scheduler failure (no session at all) produces no log entry; detectable only by an unchanged gallery in the morning. Accepted residual risk — Cowork sidebar shows the session history if it needs diagnosing.
- Credits: 60 on hand ≈ 2 weeks at nightly cadence; Recraft Basic ($10/mo, 1000 credits) needed for sustained operation.

## Out of scope

- Any write outside `concepts/` (no `client/src`, no `SlideRenderer.jsx`, no theme files, no schema).
- Approval, shipping, or promotion of prototypes — human-only.
- The 10 everyday slide transitions and the persistent ambient layer — untouched.
- Multi-prototype nights, video generation, Magnific (dropped), Adobe connector in the nightly loop — Adobe's one contribution is a **one-time, daytime asset bake**: the shared tileable grain texture (`image_add_grain`/`image_apply_halftone`) stored in `concepts/assets/` and reused by every prototype; nightly grain needs are met by inline SVG `feTurbulence` (static filter, GPU-legal). Adobe tools otherwise stay for daytime flyer/social work.
- Vercel/phone serving of the gallery; email digests (revisit only if the gallery-checking habit fails).
